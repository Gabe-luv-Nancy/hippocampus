## 一、设计目标与假设

**目标用户画像：**
- 使用 OpenClaw 的 AI 助手管理复杂多文件项目的 Operator
- 高频使用 CLI，偶尔使用 Web UI 做可视化审查
- 需要清晰掌握任务状态、文件引用状态、任务依赖关系

**核心假设：**
- Registry 总表位于 `scripts/task_schema/registry.yaml`，由 ED（Executive Director）维护
- 前端框架选用 **Streamlit**（与现有 `streamlit_app.py` 一致，降低迁移成本）
- 所有操作均通过 Python 脚本透传 CLI，不直接操作 YAML（避免人工编辑错误）

## 二、用户交互流程（完整用户旅程）

### 流程概览

```
[新建任务] → [填写表单] → [Schema 预校验] → [写入 registry] → [任务激活]
     ↓
[查看任务] → [更新状态] → [标记 file_refs verified] → [申请 done] → [终审校验] → [完成]
     ↓
[查看依赖] → [发现阻塞] → [处理前置任务] → [解锁]
```

### 2.1 新建任务流程

```
用户执行：/hippo task new "HIPPO-20260422-001" "实现 Schema 前端"
                    ↓
          Streamlit 表单打开
                    ↓
          自动填充 fields：
          ├─ id: HIPPO-20260422-001（锁定）
          ├─ title: "实现 Schema 前端"（预填，可改）
          ├─ status: proposed（默认）
          ├─ assignee: [下拉选择]
          ├─ phase: [下拉选择]
          ├─ file_refs: [空列表]
          └─ depends_on: [空列表]
                    ↓
          点击「提交」
                    ↓
          本地 Validator 预检查
          ├─ id 格式正确？→ 报错提示
          ├─ title ≤ 100字？→ 报错提示
          └─ assignee 在白名单？→ 报错提示
                    ↓
          预检通过 → 写入 registry.yaml
                    ↓
          界面反馈：「✅ 任务已创建」
                    ↓
          自动刷新任务列表（高亮新任务）
```

### 2.2 标记 file_refs verified 流程

```
用户选中任务：HIPPO-20260422-001
                    ↓
          进入「任务详情页」
                    ↓
          file_refs 区域展示：
          ┌─────────────────────────────────────┐
          │ 📎 文件引用（共 2 项）                 │
          │                                     │
          │ 1. [ ] /mnt/x/CLABIN/HIPPO/.../x.py  │
          │      路径不存在                     │
          │                                     │
          │ 2. [✅] /mnt/x/CLABIN/HIPPO/.../y.md │
          │      已验证                          │
          └─────────────────────────────────────┘
                    ↓
          用户点击路径旁的 [验证] 按钮
                    ↓
          后台执行：os.path.exists(path)
                    ↓
          存在 → [✅ 路径有效]，verified 标记 true
          不存在 → [❌ 路径无效]，提示用户修正
                    ↓
          所有 file_refs.verified == true？
          └─ 是 → "申请完成"按钮解锁
```

### 2.3 申请完成（status=done）流程

```
用户点击「申请完成」
                    ↓
          Schema 强制校验层触发
                    ↓
          ┌─ 所有 file_refs[].verified == true?
          │     └─ 否 → 【拒绝】显示未验证文件列表
          │
          └─ 所有 depends_on 任务 status == done?
                └─ 否 → 【拒绝】显示阻塞依赖任务
                    ↓
          全部通过
                    ↓
          写入 registry.yaml，status 更新为 done
                    ↓
          界面反馈：
          ┌─────────────────────────────────┐
          │ ✅ 任务 HIPPO-20260422-001 已完成 │
          │    完成时间：2026-04-22 11:45    │
          └─────────────────────────────────┘
                    ↓
          刷新依赖链视图，相关下游任务自动解锁
```

### 2.4 查看依赖关系流程

```
用户打开「依赖图」视图
                    ↓
          展示全局任务依赖有向图
          （横向时间轴 or 树状结构）
                    ↓
          节点颜色：
          🟢 done    ← 绿色，完成
          🟡 in_progress ← 黄色，进行中
          🔴 proposed   ← 红色，尚未开始
          ⬜ blocked    ← 灰色，被依赖阻塞
                    ↓
          点击节点 → 弹出该任务的快捷详情卡
                    ↓
          发现灰色节点 → 提示「该任务依赖 XXX，XXX 尚未完成」
```

## 七、技术实现笔记（For bes）

### 7.1 推荐技术栈

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **前端 UI** | Streamlit | 已有 `streamlit_app.py`，快速开发，支持 Markdown 渲染，天然支持表单和状态管理 |
| **依赖图渲染** | Graphviz + matplotlib | 纯 Python，轻量，无需 JS 依赖，适合 CLI 调用 |
| **后端数据层** | PyYAML + Python 标准库 | 与现有 `validator.py` / `registry.py` 一致 |
| **文件监控** | watchdog（可选） | 监听文件变化，自动提示刷新 |
| **Web 部署** | Streamlit Community Cloud / 本地 `streamlit run` | 支持一键部署，分享链接 |
